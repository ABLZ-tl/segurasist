# Audit Report — Batches (A3)

## Summary

Pipeline de carga masiva en buen estado general: parser/validator separados, contextos ricos, suite de tests sólida (parser, validator, curp-checksum, workers). El parser usa un mini RFC 4180 propio bien justificado y el detector de magic-bytes (L4) es defensivo y unitariamente cubierto. Sin embargo aparecen 3 bugs lógicos no triviales en la fase de procesamiento async + confirm: (1) deduplicación intra-archivo se aplica por chunk de 500 filas (rompe detección cross-chunk), (2) `confirm()` deja los contadores de la fase de validación en `rowsOk/rowsError` y el worker los incrementa encima → marcará `completed` tras la primera creación, (3) `MessageDeduplicationId` enviado a colas SQS standard (fallaría en AWS real). Otros hallazgos cubren observability, performance del send loop secuencial, y un atajo en `queueUrlForCreations` por env-schema incompleto.

## Strengths

- **Separation of concerns**: Parser (string-only), Validator (puro, recibe `ValidationContext`), Service (DB + S3 + SQS) están bien delimitados; el validator no toca DB.
- **CURP checksum SEGOB** (`curp-checksum.ts`): documenta el algoritmo paso a paso, charset oficial, casos `pre-1999`, y exporta `computeCurpChecksum` para fixtures deterministas. El test (`curp-checksum.spec.ts`) cubre 12 prefixes × {válido, inválido, lower-case, charset fuera de tabla, length ≠18, no-numérico, null/undefined}.
- **Magic bytes**: `detectFileType` rechaza ZIPs genéricos sin `xl/workbook.xml` (MAC-002 vs JAR/APK), maneja BOM UTF-8, y tiene cobertura E2E con un EXE renombrado (`batches.e2e-spec.ts:111`).
- **Idempotencia de `LayoutWorkerService`**: `batchError.deleteMany` antes de re-procesar (línea 125); errores de parser caen en `PARSE_ERROR` row-level y se persisten.
- **Idempotencia de `InsuredsCreationWorkerService`**: atrapa `Prisma P2002` y considera la creación exitosa, buscando el `insuredId` existente para emitir `insured.created` (línea 115-130).
- **Mini parser CSV propio**: justificación clara en el header (ámbito reducido, evita dep ESM, control encoding); cubre quoting `""`, `\r\n`, BOM, comas dentro de comillas, rechazo de Latin-1.
- **Levenshtein O(n) memoria**: rotación de buffers, comentarios sobre BK-tree y Damerau como upgrade futuro. `topKByLevenshtein` ordena por distancia con tie-breaker alfabético.
- **Yield al event loop entre chunks** (`layout-worker.service.ts:154`): `setImmediate` previene starvation cuando se valida 10k filas.
- **Eventos**: `insured-events.ts` y `certificate-events.ts` con discriminated unions, helpers `build*Event` y comentarios sobre owner / consumer.
- **Worker gating**: `WORKERS_ENABLED=false` por default + `prismaBypass.isEnabled()` previenen log-spam en test/CI.

## Cross-cutting concerns

- **A3 → A4 (Certificates)**: `InsuredsCreationWorkerService` emite `insured.created` a `SQS_QUEUE_PDF` (línea 142). El consumer (PDF worker) debe ser idempotente porque el `dedupeId` `${insuredId}:created` se ignora en cola standard (issue A3-03). **Append al feed**.
- **A3 → A2 (Multi-tenant/RLS)**: `LayoutWorkerService` usa `PrismaBypassRlsService` y filtra manualmente por `tenantId` en `buildContext` (línea 205, 213). `InsuredsCreationWorkerService` también. Si A2 cambia el patrón de bypass, ambos workers necesitan actualización.
- **A3 → A6 (Audit/Throttler)**: El controller usa `@TenantThrottle({ ttl: 60_000, limit: 100 })` para el upload. Si A6 redefine la decoradora, verificar.
- **A3 → A9 (DevOps/IaC)**: Las colas SQS son standard pero el código asume FIFO (issue A3-03). Decisión arquitectónica conjunta con A9 para Sprint 5 (Terraform queue defs).
- **Test signal**: `batches-flow.spec.ts` afirma ser integration pero corre in-memory; revisar con A10 (Tests/DX) para naming + categorización.

## Recommendations Sprint 4

1. **Fix A3-01 + A3-02** (bugs Critical): pre-computar `findIntraFileDuplicates(rows)` antes del loop de chunks; resetear `rowsOk/rowsError` en `confirm()` o agregar columnas `processedRows/successRows` al schema. Cubrir con un nuevo integration test que simule chunks cross-boundary y un confirm → 5 creations → completed.
2. **Decidir FIFO vs standard SQS** (A3-03): si la idempotencia P2002 es suficiente, drop `MessageDeduplicationId` del `SqsService.sendMessage`. Si se quieren colas FIFO, refactorizar bootstrap + Terraform Sprint 5 + agregar `MessageGroupId` (requerido en FIFO).
3. **Env schema completo** (A3-04): añadir `SQS_QUEUE_INSUREDS_CREATION` (y posiblemente `SQS_QUEUE_EVENTS` para `batch.preview_ready/batch.completed`). Eliminar el `string.replace` ad-hoc.
4. **Batch `SendMessage`** (A3-05): usar `SendMessageBatch` (10 msgs/call) en `confirm` para procesar 10k filas en ~1000 calls vs 10000.
5. **Cleanup A3-06 + A3-07**: simplificar la lógica de `suggestions`, extraer `stripDiacritics` a un util compartido, mover `MAX_ROWS`/`MAX_BYTES` a env.
