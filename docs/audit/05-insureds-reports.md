# Audit Report — A5 Insureds + Coverages + Packages + Reports

## Summary

A5 cubre el dominio de negocio core: `insureds` (CRUD + 360° + portal self + export pipeline + cursor pagination), `coverages` (upsert atómico + adapter user-facing → DB enum), `packages` (CRUD + archive con guard de insureds activos), `reports` (dashboard KPIs + volumetry + cache Redis 60s), `claims` (createForSelf portal + type mapping), `users` (cursor + RBAC superadmin) y `tenants` (listActive). Implementación generalmente sólida: scoping polimórfico (`platformAdmin` vs RLS), `Promise.allSettled` en find360, defense-in-depth en findSelf/findExport, anti-enumeration consistente (404 en lugar de 403), audit fire-and-forget. La cobertura unitaria es densa (~70 unit tests + integration). Los issues principales son operacionales: SQS dedupeId silenciosamente ignorado en cola estándar, ausencia de tests dedicados al `ExportRateLimitGuard`, N+1 controlado en `coveragesForSelf` (aceptable pero documentar), y un cast `as unknown as Prisma.InsuredWhereInput` que ya es obsoleto tras la migración de `cognitoSub`.

## Strengths

1. **Scope polimórfico consistente** — `InsuredsScope` / `PackagesScope` / `CoveragesScope` / `ReportsScope` siguen el mismo patrón (`platformAdmin`, `tenantId?`, `actorId?`) y enrutan a `prisma` o `bypass.client` con logging del bypass para auditabilidad (insureds.service.ts:255-264, packages.service.ts:86-95, reports.service.ts:63-72).
2. **`find360` con `Promise.allSettled`** — las 5 secciones se resuelven en paralelo con tolerancia a fallo por sección; `settled()` helper centraliza el log + fallback a `[]` (insureds.service.ts:467-496, 1037-1042). Test cubre el caso (insureds.service.spec.ts:426).
3. **Defense-in-depth** — `findSelf` y `createForSelf` chequean `user.role === 'insured'` antes del lookup (insureds.service.ts:665-667, claims.service.ts:77-79); `findExport` revalida `row.tenantId === tenant.id` aún tras RLS (insureds.service.ts:961).
4. **Anti-enumeration consistente** — 404 en lugar de 403 cuando el row pertenece a otro tenant (find360, findSelf, findExport, createForSelf). Comentado explícitamente en cada handler.
5. **Cursor opaco bien encapsulado** — `cursor.ts` separado, base64url, decode tolerante (devuelve null en lugar de 400) y testeado (round-trip + corruption + missing fields). Patrón replicado en `users/cursor.ts`.
6. **`PackagesService.archive` con guard transaccional** — bloquea si hay insureds activos (`_count.insureds > 0`) con `ConflictException` (packages.service.ts:240-256), test cubre el caso (packages.service.spec.ts:165).
7. **`upsertForPackageWithTx`** — variante que reusa una transacción ya abierta por el caller, evita transacciones anidadas y mantiene atomicidad delete-then-insert (coverages.service.ts:106-141).
8. **Cache Redis dashboard** — TTL 60s con fallback graceful: si Redis falla en `get` o `set` no rompe, sólo loguea (reports.service.ts:378-394). Cache key `_global_` para superadmin sin tenantId evita mezcla.
9. **Volumetry server-side** — `date_trunc('week', ...)` raw SQL agrupa server-side, no client (reports.service.ts:291-308). Devuelve siempre 12 buckets (rellenando con 0).
10. **Endpoints `me` declarados antes de `:id`** — handler order correcto en controller (insureds.controller.ts:77, 84) con comentario explicativo del por-qué (evita `ParseUUIDPipe` BadRequest).
11. **Coverage storage adapter documentado** — JSON envelope `{kind, unit, description}` en `description` con mapping a buckets DB `consultation`/`pharmacy`. Razón documentada en cabecera del archivo (dto/coverage-storage.ts:1-25).
12. **Hard cap exportable** — `EXPORT_ROW_HARD_CAP = 200_000` defensivo contra megacrayón PII; el worker chunkea en batches de 5k para evitar OOM (reports-worker.service.ts:332-393).

## Cross-cutting concerns (afectan a otras áreas)

- **A5↔A6 audit propagation**: `find360`, `exportRequest`, `createForSelf` invocan `auditWriter.record` pero pasan `traceId` extraído manualmente del `req.id`. El patrón se repite verbatim en 3 sitios (insureds.controller.ts:115-119, 173-177; insureds.service.ts:631-637). Recomendación: A6 expone `@AuditCtx()` decorator + un interceptor automático para handlers anotados.
- **A5↔A3 batches**: el cursor opaco `(createdAt, id)` está duplicado en `modules/insureds/cursor.ts` y `modules/users/cursor.ts` con la misma firma. Si A3/batches expone listados paginados también, considerar promover a `common/pagination/cursor.ts` (shared lib).
- **A5↔A4 certificates**: `find360` lee `certificates` por `insuredId` con `version desc`; A4 emite los certs. Aceptable acoplamiento por FK pero el shape `{id,version,issuedAt,validTo,status,hash,qrPayload}` (insureds.service.ts:601-609) duplica parcialmente lo que A4 expone — verificar que ambos contratos no diverjan.
- **A5↔A9 SQS infra**: A5-01 (deduplication broken) requiere que A9 (Terraform/IaC) renombre la cola a `.fifo` o cambie estrategia. Comunicar al feed compartido.
- **A5↔A6 throttler kill switch**: `ExportRateLimitGuard` y `@Throttle` global comparten `THROTTLE_ENABLED`. A6 debería garantizar que el flag esté `true` en prod via CI.

## Recommendations Sprint 4

1. **Cerrar A5-01 (SQS dedup)**: convertir `reports-queue` a FIFO + `MessageGroupId=tenantId` o implementar idempotencia DB con constraint compuesto. Es un High que rompe la garantía documentada del pipeline.
2. **A5-03 — tests para `ExportRateLimitGuard`**: 5 casos mínimos (cap excedido, bajo cap, kill switch, sin tenant, bypass desabilitado). Bloquea evidencia de cumplimiento del cap PII anti-abuse.
3. **A5-06 — extraer `buildInsuredsWhere` compartido**: reducir 3 copias del filter logic (service.list, service.buildExportWhere, worker.queryInsureds). Reduce drift risk antes de que el FE pida un filtro nuevo.
4. **A5-04 — `coveragesForSelf` con `groupBy`**: consolidar a una sola query agregada. Trivial fix con beneficio observable cuando un paquete crezca a 15+ coberturas.
5. **A5-02 — limpiar el cast obsoleto en `findSelf`/`createForSelf`**: la migración `cognitoSub` ya está applied; el `as unknown as Prisma.InsuredWhereInput` es deuda residual.
6. **A5-08 — decisión sobre `coverageConsumedPct.trend`**: implementar trend real (requiere snapshot 30d en tabla nueva, candidato Sprint 5) o cambiar contrato a `trend?: number` y dejar `0` como señal de "no aplicable" hasta entonces.
