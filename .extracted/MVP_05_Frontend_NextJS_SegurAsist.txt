1. Resumen y stack frontend
El frontend de SegurAsist es un monorepo Next.js 14 con App Router que aloja dos aplicaciones independientes pero coexistentes: el portal administrativo (admin.segurasist.mx) y el portal del asegurado (portal.segurasist.mx). Ambas comparten un design system y un cliente API tipado, y se despliegan en AWS Amplify Hosting con SSR + RSC + ISR según ruta.

Mobile-first (viewport target 360 px). 1 columna principal en mobile, 2 columnas en tablet/desktop. Toda la navegación cabe en una bottom-nav (Home, Coberturas, Certificado, Ayuda) en mobile y top-nav en desktop.
Tipografía grande (16 px base mobile, 18 px desktop), contraste alto (verde de vigencia ≥ 7:1), botones de tap ≥ 44 px.

/login: input CURP (placeholder explicativo), select de canal (Email / SMS), botón "Enviar código".
/otp: input de 6 dígitos (autoFocus, paste-friendly), botón "Verificar", link "Reenviar código" (deshabilitado 30 s).

6. Autenticación y seguridad en el cliente
6.1 Flujo OAuth + manejo de tokens
	•	Login redirige a Cognito Hosted UI (SAML/OIDC) o usa flujo CUSTOM_CHALLENGE para OTP.
	•	Callback API route (/api/auth/callback) recibe code, intercambia por tokens en server, fija cookies HttpOnly Secure SameSite=Lax con accessToken (15 min) y refreshToken (7 d).
	•	Middleware Next.js (middleware.ts) en cada request server: valida JWT contra JWKS (cache); si vencido, refresca usando refreshToken; redirige a /login si imposible.
	•	API requests desde cliente usan fetch wrapper que añade Authorization desde la cookie via /api/proxy (server-to-server al backend).
	•	Logout: borra cookies + GET /v1/auth/logout + redirect a /login.

6.3 Manejo de errores y feedback
	•	Toast de error con mensaje humano (mapeado de códigos backend); incluye traceId si severity ≥ error.
	•	Boundary global captura errores de render y muestra fallback con botón "Reportar" (envía traceId + screenshot opcional).
	•	Estados loading siempre con Skeleton o Spinner (no pantalla en blanco).
	•	Estados vacíos siempre con EmptyState (icono, mensaje, CTA).

7. Estado de datos y data fetching
7.1 Reglas
	•	Componentes Server (RSC) por default; pasan a Client Component sólo si necesitan interactividad o hooks.
	•	Datos públicos cacheables: fetch nativo con revalidate.
	•	Datos por usuario: TanStack Query con staleTime 60 s para listados, 0 para mutaciones.
	•	Mutaciones: optimistic updates donde aplique; invalidación selectiva por queryKey.
	•	WebSocket / SSE no para MVP; polling 2 s para batches en proceso.
7.2 Cliente API tipado
openapi-typescript genera tipos de OpenAPI del backend. Cliente fetch wrapper:
export async function api<T>(path: string, init?: RequestInit): Promise<T> {
  const traceId = crypto.randomUUID();
  const res = await fetch(`/api/proxy${path}`, {
    ...init,
    headers: {
      'content-type': 'application/json',
      'x-trace-id': traceId,
      ...(init?.headers ?? {})
    }
  });
  if (!res.ok) throw await ProblemDetailsError.from(res, traceId);
  return res.json();
}
Hooks por recurso (packages/api-client/hooks/insureds.ts):
export const useInsureds = (params: ListParams) =>
  useQuery({
    queryKey: ['insureds', params],
    queryFn: () => api<InsuredsList>(`/v1/insureds?${qs(params)}`),
    staleTime: 60_000
  });

export const useCreateInsured = () => {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (dto: CreateInsuredDto) =>
      api<Insured>('/v1/insureds', { method: 'POST', body: JSON.stringify(dto) }),
    onSuccess: () => qc.invalidateQueries({ queryKey: ['insureds'] })
  });
};

8. Accesibilidad (WCAG 2.1 AA)
	•	Contraste mínimo 4.5:1 texto normal, 3:1 texto grande; verde vigencia 7:1 (mejor que mínimo).
	•	Foco visible en todos los elementos interactivos (outline de 2 px accent color).
	•	Navegación completa por teclado (Tab, Shift+Tab, Enter, Esc, flechas en menus/tabs).
	•	ARIA labels en iconos sólos; aria-live="polite" para toast no críticos, "assertive" para críticos.
	•	Forms: label asociada por htmlFor; mensajes de error vinculados con aria-describedby.
	•	Tablas: scope="col" en th; resúmenes accesibles para tablas complejas.
	•	Skip link "Saltar al contenido" al inicio del body.
	•	Contenido oculto visualmente pero accesible para screen readers via clase sr-only.
	•	prefers-reduced-motion respetado: animaciones reducidas a fade simples.
	•	Audit Lighthouse en CI; falla si Accessibility < 90 en portal asegurado, < 85 en admin.

11. Internacionalización
Inicial: es-MX. Arquitectura con next-intl preparada para añadir en-US sin reescribir componentes.

	•	Formateo de moneda Intl.NumberFormat MXN.
	•	CURP/RFC NUNCA traducidos; mantienen mayúsculas.
	•	Pluralización: ICU MessageFormat ("{count, plural, one {1 asegurado} other {# asegurados}}").
