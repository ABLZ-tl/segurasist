1. Resumen arquitectónico
SegurAsist se construye como una plataforma SaaS multi-tenant 100% sobre AWS, optimizada para un cliente ancla (Hospitales MAC) y diseñada para replicarse a SNTE, CATEM, Walmart y otros sin redesarrollo. La arquitectura privilegia servicios gestionados (managed services) para minimizar la sobrecarga operativa de un equipo pequeño y maximizar el cumplimiento heredado del proveedor cloud.
La región primaria es us-east-1 (N. Virginia) con réplica de DR cross-region en us-east-2 (Ohio). El componente de cómputo principal es AWS App Runner ejecutando un backend NestJS (TypeScript). La persistencia es Amazon RDS PostgreSQL 16 Multi-AZ con Row-Level Security para aislar tenants. El frontend es Next.js 14 con App Router desplegado en AWS Amplify Hosting. La identidad y SSO se gestiona con Amazon Cognito User Pools. Las cargas asíncronas (procesamiento de layouts, generación PDF, envío de correos) viven en AWS Lambda activadas por Amazon SQS y EventBridge. Los archivos (PDF de certificados, layouts subidos, exports) se almacenan en Amazon S3 con cifrado KMS y, en buckets de auditoría, S3 Object Lock en modo Compliance.
1.1 Pilares de la arquitectura
	•	Multi-tenant aislado por construcción: tenant_id en cada tabla + RLS PostgreSQL + tests CI cross-tenant + prefijos de bucket separados.
	•	Serverless-first donde reduce TCO: SQS+Lambda en lugar de Redis+BullMQ; App Runner en lugar de EKS; Amplify en lugar de ECS para el frontend.
	•	Seguridad por default: TLS 1.3 forzado, AES-256 con CMK rotación anual, MFA obligatorio admin, WAF AWS Managed Rules, GuardDuty 24/7, logs inmutables S3 Object Lock 24 m.
	•	Observabilidad estructurada: trace_id propagado entre frontend, API y workers; logs JSON; alarmas CloudWatch con on-call PagerDuty.
	•	IaC obligatorio: 0 cambios manuales en consola para recursos productivos; Terraform versionado en GitHub.
	•	Migración futura sin re-código: si MAC exige residencia mx-central-1, se replantea sólo Terraform; Cognito y SES tienen workarounds documentados.

4.2 Estrategia de aislamiento multi-tenant
Dos capas de defensa redundantes:
	•	Application: cada query lleva tenant_id resuelto desde el JWT (claim custom:tenant_id) por middleware que rechaza la petición si falta. ORM (Prisma) intercepta y agrega tenant_id en where automáticamente.
	•	Database: PostgreSQL Row-Level Security activa con políticas por tabla. Antes de cada query la sesión ejecuta SET app.current_tenant = '<uuid>'. Las políticas USING comparan tenant_id contra ese setting; si no coincide, el row no es visible NI a SELECT, INSERT, UPDATE ni DELETE.
Tests CI dedicados intentan en cada PR consultar recursos del tenant B autenticados como tenant A: el resultado esperado es 403 / 0 filas. Si pasan filas, la build falla.
Ejemplo de política RLS para insureds:
ALTER TABLE insureds ENABLE ROW LEVEL SECURITY;
CREATE POLICY insureds_tenant_iso ON insureds USING (tenant_id::text = current_setting('app.current_tenant', true));
4.3 Estrategia de migraciones
	•	Tooling: Prisma Migrate (sincroniza con Prisma schema y emite SQL idempotente).
	•	Política: migraciones forward-only en producción; rollback se maneja con migración compensatoria.
	•	Despliegue: la migración se ejecuta automáticamente al inicio del deploy en App Runner mediante script pre-start; falla del deploy si la migración falla.
	•	Datos: migraciones de datos costosas se ejecutan como Lambda one-off con feature flag.

6. Vista de procesos
6.1 Flujo: carga masiva de asegurados
	•	UI sube archivo a endpoint POST /v1/batches con multipart/form-data.
	•	API valida tamaño (≤25 MB), MIME y firma del usuario; sube a S3 (uploads/{tenant}/{batchId}.xlsx) con SSE-KMS.
	•	API persiste batches con estado validating; emite mensaje SQS al worker layout.
	•	Lambda Worker Layout descarga el archivo, parsea con SheetJS, valida cada fila (CURP/RFC/paquete/duplicados), escribe batch_errors si aplica.
	•	Lambda actualiza batches con conteos y estado preview_ready. UI hace polling al endpoint GET /v1/batches/{id}.
	•	Operador confirma desde UI: POST /v1/batches/{id}/confirm. API encola N mensajes (uno por fila válida) en SQS de creación.
	•	Lambda crea cada insured (transacción), publica evento EventBridge insured.created.
	•	EventBridge dispara Lambda Worker PDF para cada insured.created.
	•	Worker PDF renderiza plantilla, sube PDF a S3 (certificates/{tenant}/{insuredId}/{version}.pdf), persiste certificate.
	•	Publica evento certificate.issued; otra Lambda envía correo con SES; SES emite eventos delivered/bounced que llegan a Lambda Worker Email vía SNS.
	•	Operador ve el lote completo en dashboard; cada row del lote tiene estado de certificado y email.
6.2 Flujo: login asegurado con OTP
	•	UI portal pide CURP.
	•	API verifica que CURP existe en insureds vigente; pide a Cognito generar OTP a email/SMS según preferencia.
	•	Cognito envía OTP; UI muestra pantalla de captura (TTL 5 min, 5 intentos).
	•	Asegurado ingresa OTP; Cognito emite JWT con claims (sub, email, custom:insured_id, custom:tenant_id).
	•	UI guarda token (cookie HttpOnly Secure SameSite=Lax).
	•	Llamadas posteriores mandan Authorization: Bearer; API valida firma JWT contra JWKS de Cognito y aplica RLS context.
6.3 Flujo: generación de reporte mensual
	•	Cron EventBridge dispara Lambda Worker Reportes el día hábil 1 a las 09:00 (zona MX).
	•	Lambda agrega datos de altas, bajas, certificados, utilización del mes anterior por tenant y entidad.
	•	Genera XLSX con SheetJS y PDF con Puppeteer; firma con metadatos del CISO.
	•	Sube ambos a S3 (exports/{tenant}/reports/{yyyy-mm}/conciliation.xlsx).
	•	Envía email a destinatarios configurados por tenant con URL presigned (TTL 30 días).
	•	Notifica a Roy vía Slack webhook si está configurado.

7. Architecture Decision Records (ADRs)
Esta sección consolida los ADRs ya cerrados en el Stack Tecnológico Definitivo y agrega los específicos del MVP.
7.1 ADR-001 — AWS como único proveedor cloud
Campo
Valor
Status
Aceptado (Stack Definitivo v1.0, abril 2026)
Contexto
Necesitamos cloud provider único para reducir DPAs, paneles, contratos.
Decisión
100% AWS. Region primaria us-east-1, DR us-east-2.
Consecuencias
Stack cohesivo (Cognito + SES + App Runner). Lock-in moderado mitigado por usar estándares (Postgres, OAuth, S3 API).
Alternativas evaluadas
GCP — descartado: diferencia de costo marginal, menor disponibilidad de talento MX, DPAs separados para servicios afines.
7.2 ADR-002 — App Runner sobre ECS/EKS
Campo
Valor
Status
Aceptado
Contexto
MVP con tráfico moderado y equipo pequeño no justifica K8s/cluster propio.
Decisión
AWS App Runner para el backend NestJS.
Consecuencias
Cero overhead de cluster; auto-scale; billing por uso. Limitación: 1 contenedor por servicio (suficiente). Migración futura a ECS Fargate trivial.
Alternativas
ECS Fargate (más control, más overhead), EKS (sobreingeniería para MVP), Lambda containers (límite 15 min, no apto para REST sostenida).
7.3 ADR-003 — RDS PostgreSQL Multi-AZ vs Aurora Serverless v2
Campo
Valor
Status
Aceptado
Contexto
Carga 24/7 con conexiones persistentes y necesidad de RLS bien documentado.
Decisión
RDS PostgreSQL 16 Multi-AZ (db.t4g.small) con pgAudit y CIS Parameter Group.
Consecuencias
Costo predecible (~USD 55/mes); RLS first-class. Aurora SLS evaluable cuando volumen lo justifique.
Alternativas
Aurora Serverless v2: USD 87/mes mínimo; ventaja por bursts no aplica.
7.4 ADR-004 — Cognito sobre Auth0
Campo
Valor
Status
Aceptado
Contexto
Necesitamos SAML, OIDC, MFA, SCIM, OTP, multi-tenant (admin pool + insured pool).
Decisión
Amazon Cognito User Pools. Pool separado por audiencia (admin vs asegurado).
Consecuencias
Costo ~0 hasta 50k MAU; integración nativa AWS; menor pulido UI mitigado con Amplify UI.
Alternativas
Auth0: USD 240/mes desde 0; ventaja UI no compensa costo y proveedor extra en DPA.
7.5 ADR-005 — SQS + Lambda sobre Redis + BullMQ
Campo
Valor
Status
Aceptado
Contexto
Cargas asíncronas: parsing layouts, generación PDF, envío emails, reportes.
Decisión
SQS + Lambda. ElastiCache Redis Serverless solo para sesión, throttle, OTP cache.
Consecuencias
Pay-per-invocation; sin workers 24/7. Cold start mitigado con provisioned concurrency en Worker PDF.
Alternativas
BullMQ: requiere worker container 24/7 + Redis dedicada (~USD 15/mes mínimo).
7.6 ADR-006 — Next.js + Amplify Hosting sobre Vercel/ECS
Campo
Valor
Status
Aceptado
Contexto
Frontend SSR con CDN global, build CI/CD integrado, ambiente AWS único.
Decisión
Next.js 14 App Router + AWS Amplify Hosting.
Consecuencias
Build automatizado desde GitHub, CDN CloudFront detrás, costo predecible.
Alternativas

7.7 ADR-007 — RLS PostgreSQL para aislamiento multi-tenant
Campo
Valor
Status
Aceptado
Contexto
Aislamiento crítico para 88.7% cumplimiento V2 y para preparar SNTE/CATEM/Walmart.
Decisión

10. Estrategia de evolución
Decisiones explícitas para evitar deuda accidental:
	•	Cuando el costo Datadog/Grafana se justifique (>USD 200/mes en CW logs), migrar logs a Grafana Cloud Free + retener métricas en CloudWatch.
	•	Si MAC requiere residencia mx-central-1, replantear sólo Terraform; código no cambia.
	•	Si llega cliente enterprise con exigencia de aislamiento de schema, activar feature flag tenant.dedicated_schema y crear schema con Prisma Migrate.
	•	Si volumen >250k MAU, migrar Cognito a Cognito Identity Federation + AWS IAM con roles temporales para acceso S3 directo.
	•	Si tráfico picos >5x base, migrar App Runner a ECS Fargate con auto-scaling más fino.
	•	Si llegamos a SOC 2 Tipo II y necesitamos audit trail externo, agregar AWS QLDB para log inmutable enriquecido.
