1. Resumen y stack backend
La API de SegurAsist es un servicio NestJS 10 + TypeScript desplegado como contenedor en AWS App Runner (1 vCPU, 2 GB) con escalado automático. Persistencia en RDS PostgreSQL 16 vía Prisma ORM. Identidad delegada a Amazon Cognito. Trabajos asíncronos en AWS Lambda activados por Amazon SQS y EventBridge. Storage en Amazon S3. Cache, sesiones y throttling en ElastiCache Redis Serverless.

4. Autenticación y autorización
4.1 Flujo Cognito
	•	Frontend (admin) inicia OAuth Authorization Code Flow contra Cognito. Para asegurados, flujo CUSTOM_CHALLENGE (CURP + OTP).
	•	Cognito emite Access Token (JWT, 15 min) e ID Token (JWT, 15 min) y Refresh Token (7 días).
	•	Frontend almacena access token en cookie HttpOnly Secure SameSite=Lax.
	•	Cada request al API lleva Authorization: Bearer.
	•	AuthGuard valida firma JWT contra JWKS público de Cognito (cacheado 24h con stale-while-revalidate).
	•	AuthGuard inyecta CurrentUser y Tenant en el request a partir de claims (sub, email, custom:tenant_id, custom:role, scope).
	•	RolesGuard verifica que role esté autorizado por @Roles() decorator del handler.
	•	PrismaService middleware fija session_replication_role + SET app.current_tenant para activar RLS.

@Controller({ path: 'insureds', version: '1' })
@UseGuards(JwtAuthGuard, RolesGuard)
export class InsuredsController {
  @Get(':id')
  @Roles('admin_mac', 'operator', 'admin_segurasist', 'supervisor')
  @Scopes('read:insureds')
  async getById(@Param('id') id: string, @Tenant() tenant: TenantCtx) {
    return this.insureds.findOne(id, tenant);
  }
}
4.4 PrismaService con RLS context
@Injectable({ scope: Scope.REQUEST })
export class PrismaService extends PrismaClient {
  constructor(@Inject(REQUEST) req: FastifyRequest) {
    super({ log: ['warn', 'error'] });
    const tenantId = (req as any).tenant?.id;
    this.$use(async (params, next) => {
      if (!tenantId) throw new ForbiddenException('Tenant context missing');
      await this.$queryRawUnsafe(`SET LOCAL app.current_tenant = '${tenantId}'`);
      return next(params);
    });
  }
}

5. Validación, DTOs y errores
5.1 Validación con Zod
Todos los inputs HTTP se validan con Zod a través de un pipe genérico. Los esquemas viven al lado de los DTOs en cada módulo. Ejemplo:
export const CreateInsuredSchema = z.object({
  curp: z.string().regex(/^[A-Z]{4}\d{6}[HM][A-Z]{5}[A-Z0-9]\d$/),
  rfc: z.string().regex(/^[A-Z&Ñ]{3,4}\d{6}[A-Z0-9]{3}$/),
  fullName: z.string().min(3).max(120),
  dob: z.string().date(),
  email: z.string().email().optional(),
  phone: z.string().regex(/^\+?\d{10,15}$/).optional(),
  packageId: z.string().uuid(),
  validFrom: z.string().date(),
  validTo: z.string().date(),
  beneficiaries: z.array(BeneficiarySchema).max(10).optional()
}).refine(d => new Date(d.validTo) > new Date(d.validFrom), {
  message: 'validTo debe ser posterior a validFrom', path: ['validTo']
});
export type CreateInsuredDto = z.infer<typeof CreateInsuredSchema>;

7. Workers Lambda
7.1 Worker Layout
Trigger: SQS layout-validation-queue. Payload: { batchId, tenantId, s3Key }.
	•	Descarga el archivo desde S3 con KMS decrypt.
	•	Parsea con SheetJS (xlsx); detecta hoja "asegurados" o primera.
	•	Itera filas. Por cada fila: valida con Zod schema, busca duplicados (CURP en tenant), valida package_id existe.
	•	Insert batch en transacción: actualiza counts cada 1000 filas para mostrar progreso.
	•	Marca el batch como preview_ready y publica evento batch.preview_ready.
Memoria: 1024 MB. Timeout: 5 min. Reservada concurrency: 5.
7.2 Worker PDF
Trigger: EventBridge insured.created y certificate.reissue_requested. Payload: { insuredId, tenantId, version? }.
	•	Carga datos del asegurado y plantilla (caché en memoria local con TTL 5 min).
	•	Renderiza HTML con Handlebars (datos + branding del tenant).
	•	Ejecuta puppeteer-core con Chromium layer (chrome-aws-lambda).
	•	Aplica metadata XMP + QR de verificación (URL al endpoint /v1/certificates/verify/{hash}).
	•	Sube PDF a S3 con SSE-KMS y persiste certificate.
	•	Publica evento certificate.issued.
Memoria: 2048 MB. Timeout: 30 s. Provisioned concurrency: 2 en horario laboral.
7.3 Worker Email
Trigger: EventBridge certificate.issued. Payload: { certificateId, tenantId, insuredId }.
	•	Genera URL S3 presigned (TTL 7 días).
	•	Renderiza plantilla email con branding del tenant (Handlebars + Polished).
	•	Llama SES SendEmail (Configuration Set para tracking de eventos).
	•	Persiste registro inicial email_events con estado queued.
Memoria: 256 MB. Timeout: 10 s.
7.4 Worker Reports
Trigger: SQS report-generation-queue (manual o cron EventBridge).
	•	Ejecuta queries agregadas con timeouts (≤30 s).
	•	Construye XLSX con SheetJS y PDF con Puppeteer.
	•	Sube ambos a S3 (exports/{tenant}/reports/...).
	•	Si la solicitud es interactiva, escribe URL en BD para que API responda al polling.
	•	Si es schedule, dispara Worker Email para enviar.
Memoria: 3008 MB. Timeout: 5 min.
7.5 Webhook handler SES (Lambda + SNS)
SES envía Bounce/Complaint/Delivery a un topic SNS. Una Lambda suscrita al topic verifica firma SNS, persiste email_events y emite alerta si bounce hard cruza umbral del 5% diario.

8. Capa de datos (Prisma + Postgres)
8.1 Convenciones
	•	Todas las tablas tienen tenant_id NOT NULL excepto tenants y configuraciones globales.
	•	PK uuid v4 generado por la app (no sequence).
	•	Timestamps: created_at, updated_at con default now().
	•	Soft delete: columna status enum + deleted_at nullable; nunca DELETE físico en tablas core.
	•	Índices: tenant_id en cada tabla; índices compuestos según query plan.
	•	Enums fuertes en Postgres (CREATE TYPE) y Prisma.
8.2 Snippet schema.prisma (extracto)
model Insured {
  id            String   @id @default(uuid())
  tenantId      String   @map("tenant_id")
  curp          String
  rfc           String?
  fullName      String   @map("full_name")
  dob           DateTime
  email         String?
  phone         String?
  packageId     String   @map("package_id")
  validFrom     DateTime @map("valid_from")
  validTo       DateTime @map("valid_to")
  status        InsuredStatus @default(active)
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")
  deletedAt     DateTime? @map("deleted_at")

  tenant        Tenant   @relation(fields: [tenantId], references: [id])
  package       Package  @relation(fields: [packageId], references: [id])
  certificates  Certificate[]
  claims        Claim[]
  beneficiaries Beneficiary[]
  usages        CoverageUsage[]

  @@unique([tenantId, curp], name: "insured_tenant_curp_unique")
  @@index([tenantId, status])
  @@index([tenantId, packageId])
  @@map("insureds")
}
8.3 Migraciones — práctica
	•	npx prisma migrate dev --name <nombre> en local crea la migración SQL.
	•	PR incluye la migración revisada por Tech Lead.
	•	CI ejecuta prisma migrate deploy en staging contra una réplica de prod (snapshot).
	•	Deploy a prod corre prisma migrate deploy automáticamente vía hook pre-start.
	•	Si la migración falla, App Runner no actualiza la versión y el tráfico sigue en versión anterior.
8.4 RLS — script de inicialización
-- Para cada tabla con tenant_id
ALTER TABLE insureds ENABLE ROW LEVEL SECURITY;
CREATE POLICY p_insureds_select ON insureds FOR SELECT
  USING (tenant_id::text = current_setting('app.current_tenant', true));
CREATE POLICY p_insureds_modify ON insureds FOR ALL
  USING (tenant_id::text = current_setting('app.current_tenant', true))
  WITH CHECK (tenant_id::text = current_setting('app.current_tenant', true));

11. Performance y rendimiento
	•	Conexiones BD pool: 10 por instancia App Runner; máximo 30 conexiones a RDS db.t4g.small (límite 87).
	•	Cache Redis para: lista de paquetes (TTL 5 min), claims agregados (TTL 1 min), JWKS Cognito (TTL 24 h).
	•	Listados con cursor-based pagination (no offset) para mantener performance constante.
	•	Queries con EXPLAIN ANALYZE en PR si el handler toca tablas > 100k filas.
	•	Índices propuestos: insureds(tenant_id, curp) UNIQUE, insureds(tenant_id, status), certificates(tenant_id, insured_id), audit_log(tenant_id, occurred_at DESC).
	•	Trigram (pg_trgm) para búsqueda por nombre.
	•	HTTP keep-alive habilitado y compresión gzip/brotli para responses > 1 KB.
