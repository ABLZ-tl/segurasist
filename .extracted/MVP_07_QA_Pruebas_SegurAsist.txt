1. Estrategia general
La calidad del MVP de SegurAsist se construye con una pirámide de pruebas balanceada (60% unitarios, 25% integración, 10% E2E, 5% manual exploratorio), pruebas de seguridad continuas (SAST, SCA, DAST, cross-tenant gate), pruebas de rendimiento por release y un proceso UAT con MAC en sprint 6.
QA es responsabilidad compartida: cada developer es dueño de los tests de su código; el rol QA dedicado lidera estrategia, automatiza E2E, ejecuta pruebas de seguridad y rendimiento, y valida criterios de release. La filosofía es shift-left: cada bug encontrado en staging tiene que generar un test que lo previene a futuro.
1.1 Objetivos de calidad
	•	Cobertura unitaria backend ≥70%, frontend ≥60%.
	•	100% de endpoints con tests cross-tenant (gate de PR).
	•	0 defectos Críticos/Altos abiertos al Go-Live.
	•	0 hallazgos High/Critical de SAST/SCA/DAST sin remediación al Go-Live.
	•	100% historias del PRD con E2E happy path verde.
	•	Lighthouse Accessibility ≥90 portal asegurado, ≥85 admin.
	•	Performance: API p95 ≤500 ms, portal LCP ≤2.0 s, admin LCP ≤2.5 s.

4. Gate cross-tenant — el más importante
Aislamiento multi-tenant es controles V2 "AppSec – Aislamiento Multitenant" + "IAM – RBAC/ABAC". Es el único control donde una falla pasa de Medio a Crítico instantáneamente. Por eso tiene una suite dedicada que corre en cada PR y bloquea merge.
4.1 Suite test/security/cross-tenant.spec.ts
Para cada endpoint que recibe tenant_id implícito (es decir, todos excepto /v1/auth/*), se prueba que un usuario autenticado en tenant A intentando operar sobre recurso que pertenece a tenant B obtenga 404 (preferido) o 403, NUNCA 200 ni datos.
4.2 Patrones cubiertos
	•	GET /resource/{id} con id de otro tenant → 404.
	•	GET /resource?filter=X devuelve únicamente recursos del tenant del JWT (count en BD vs count en respuesta).
	•	PATCH /resource/{id} con id de otro tenant → 404 + sin modificación BD.
	•	DELETE /resource/{id} con id de otro tenant → 404 + sin modificación BD.
	•	Cross-tenant via filtros: ?tenantId=B en query string ignorado/rechazado.
	•	Cross-tenant via SQL injection: payloads conocidos OWASP rechazados.
	•	RLS efectiva: query directa con usuario app sin SET app.current_tenant devuelve 0 filas.
	•	Bucket S3 prefijo de tenant respetado en URLs presigned.
Si cualquier test pasa filas/ids del otro tenant, la build falla en rojo. PagerDuty P1 si llega a main de algún modo.

5. Pruebas de seguridad continuas
5.1 SAST — Semgrep
	•	Ruleset: p/owasp-top-ten + p/typescript + p/javascript + reglas custom para Prisma sin tenantId.
	•	Falla PR si findings High o Critical.
	•	Findings Medium → ticket Linear con SLA 30 d.
	•	Falsos positivos: marcados con // semgrep-disable-next-line con justificación obligatoria.
5.2 SCA — Dependabot + npm audit + Trivy
	•	Dependabot semanal en ambos repos para npm + GitHub Actions.
	•	npm audit en CI; falla si severity ≥ high.
	•	Trivy escanea imágenes Docker en build-image; falla si CVE Critical o High sin fix disponible (excepción documentada).
	•	Renovate para Terraform providers semanal.
5.3 DAST — OWASP ZAP
	•	ZAP Baseline scan corre en cada deploy a staging contra api.staging.segurasist.mx + portal.
	•	ZAP Full scan nightly contra staging.
	•	Falla deploy si hallazgos High; Medium genera ticket.
	•	Reglas custom: chequeo de headers de seguridad, JWT en URLs, secretos en HTML.
5.4 IAST y otros (futuro)
	•	BurpSuite manual en sprint 5 por QA + CISO sobre flujos críticos.
	•	Checkov en Terraform PRs (chequeos IaC).
	•	gitleaks pre-commit hook + en CI.
	•	Pentest externo formal Q3-2026 (post Go-Live).
5.5 Pentest interno (sprint 5)
Caja blanca por equipo QA + CISO con apoyo de Backend Sr. Cobertura mínima:
	•	OWASP API Security Top 10 sobre /v1/* completo.
	•	OWASP Web Top 10 sobre admin y portal asegurado.
	•	Auth: brute force, JWT replay, JWT alg=none, kid manipulation, OTP race condition.
	•	Authz: IDOR (insureds, certificates, batches, claims), elevación de privilegios.
	•	Multi-tenant: 30 escenarios cross-tenant manuales adicionales a los automatizados.
	•	Storage: enumeración de buckets, URLs presigned forjadas, path traversal.
	•	Email: spoofing, reinjection en campos.
	•	PDF: XSS persistido en plantilla, exfiltración de cookie.
	•	WAF: bypass de rate limit, payloads SQL/XSS conocidos.

7. Datos de prueba y ambientes
7.1 Generación de datos sintéticos
	•	Faker.js con localización es-MX para nombres y direcciones.
	•	CURP/RFC generados con dígitos verificadores válidos (no extraídos de personas reales).
	•	Email/teléfono dirigidos a dominios sandbox (segurasist.test, +52 555 000 0000).
	•	Seeds reproducibles con semilla configurable; dataset de 10k, 100k, 1M para distintos escenarios.
	•	Limpieza automática nightly en dev.
7.2 Datos en staging
Staging usa anonimización del subset productivo aprobado: nombres reemplazados, CURPs falsificados, fechas desplazadas. Pipeline de scrubbing corre semanal.
7.3 Reglas absolutas
	•	PROHIBIDO copiar datos productivos a dev/staging sin scrubbing.
	•	PROHIBIDO usar emails reales en pruebas (riesgo de envío accidental por SES).
	•	PROHIBIDO compartir cookies/tokens de prod.

9.2 Flujo
	•	QA o developer descubre el defecto y lo registra en Linear con plantilla (pasos, esperado, real, evidencia, severidad propuesta).
	•	Triage diario por QA + Tech Lead define severidad final y owner.

10. Release gates
10.1 Gate de PR a main
	•	Lint + typecheck verdes.
	•	Test unit + integración + cross-tenant verdes.
	•	SAST sin findings High/Critical.
	•	SCA sin CVE High/Critical sin fix.
	•	Coverage no decrece bajo umbral.
	•	Aprobación de al menos 1 reviewer distinto al autor.
10.2 Gate de deploy a staging
	•	Build de imagen + scan Trivy verdes.
	•	Migraciones aplicables (prisma migrate diff).
	•	Smoke tests (Playwright) post-deploy verdes.
	•	DAST OWASP ZAP Baseline sin High.
	•	Lighthouse Performance/Accessibility no decrece.
10.3 Gate de Go-Live (release-ready)
	•	Todos los gates anteriores verdes.
	•	Pentest interno con 0 findings High/Critical abiertos.
	•	DR drill ejecutado y RTO/RPO confirmados.
	•	UAT firmada por MAC + Roy.
	•	Backups verificados por restauración real.
	•	Plan de rollback documentado y ensayado.
	•	Comunicación a usuarios MAC preparada.
	•	Hypercare on-call programado 48 h.
	•	Aprobación firmada Tech Lead + DevOps + QA + PO + CISO.
